# 1.0.0 Release Plan

## Overview

This plan describes the work required to ship a stable 1.0.0 release of the
`in_app_messaging` monorepo. The release covers three packages:

| Package | Current | Target |
|---|---|---|
| `in_app_messaging` | `0.0.21` | `1.0.0` |
| `in_app_messaging_drift` | `0.0.22` | `1.0.0` |
| `in_app_messaging_json_logic` | `2.3.0` | `2.3.0` (no changes) |

The strategy: **scope down to what works, harden it, then ship**. No new features
until after 1.0.0. Broken or experimental surface is removed from the public API,
not fixed in place.

---

## Phase Dependency Map

```
Phase 1 (API Surgery)
    ├── Phase 2 (Tests)
    │       └── Phase 3a (CI) ← needs tests to exist
    ├── Phase 3b (Examples)   ← uses stable API
    └── Phase 4 (Docs)        ← describes stable API
            └── Phase 5 (Release)
```

Phase 1 is the unblocking foundation. Phases 2, 3b, and 4 can be parallelized
after Phase 1 completes.

## Phase 1 — API Surgery

**Goal:** Public API contains only things that work. Pure deletion and
refactoring — no new features. After this phase `dart analyze` produces zero
errors and all currently-passing tests still pass.

**`packages/in_app_messaging/lib/src/domain/gateway/message_gateway.dart`**

- Remove abstract `setDeviceProperty(String key, String? value)` method

**`packages/in_app_messaging/lib/src/data/gateway/default_message_gateway.dart`**

> `updateDeviceProperty`) be promoted to the `ContextSource` interface for

## Phase 2 — Test Foundation

**Goal:** `in_app_messaging_drift` goes from zero functional tests to meaningful
coverage. The core package gets a gateway integration test covering the full
trigger → evaluate → markSeen path.

### 2.1 Fix `in_app_messaging_drift` dev_dependencies

**`packages/in_app_messaging_drift/pubspec.yaml`**

Add to `dev_dependencies`:
```yaml
test: ^1.25.2
sqlite3: ^2.4.0   # for NativeDatabase.memory() in VM tests
```

> Note: Drift tests using `NativeDatabase.memory()` run under `dart test` (no
> Flutter needed). This avoids a `flutter test` dependency for a package that
> contains no Flutter widgets.

### 2.2 Write Drift DAO tests

Create `packages/in_app_messaging_drift/test/dao/in_app_messaging_dao_test.dart`
using an in-memory `NativeDatabase`. Cover:

- `markSeen` inserts a row into the seen-dates table
- `getSeenEntries` returns an empty list for an unknown message id
- `getSeenEntries` returns the correct `MessageSeenEntry` after one `markSeen`
- Multiple `markSeen` calls for the same message id return multiple entries
- `markSeen` with `null` trigger and `null` triggerProperties stores nulls
- `markSeen` with non-serializable triggerProperties falls back to `null` rather
  than throwing (exercises the try/catch in `markSeen`)

### 2.3 Write Drift migration tests

Create `packages/in_app_messaging_drift/test/database/migration_test.dart`
using the already-generated schema classes in
`test/database/generated_migrations/`. Cover:

- Schema at each version matches its generated schema class (`v0`, `v1`, `v2`)
- Migration v1 → v2 adds `trigger` and `trigger_properties` columns to the
  seen-dates table

### 2.4 Write gateway integration tests

Create `packages/in_app_messaging/test/gateway_integration_test.dart`. Uses
in-memory sources only — no widget tree. Cover:

- Message returned when event matches trigger
- Message not returned when event does not match
- Message not returned when `enabled = false`
- Message not returned before `start` date
- Message not returned after `end` date
- Priority ordering: lower `priority` value wins when multiple messages match
- `OnceCondition`: returned on first call, not on second after `markSeen`
- `OncePerDayCondition`: returned on first trigger, not again same day
- The 5-minute `AsyncCache` window is respected (second call within window
  does not re-fetch from source)

### 2.5 Add `melos run analyze` script

**`pubspec.yaml`** (workspace root) — add:

```yaml
analyze:
  run: |
    melos exec --depends-on=flutter -- flutter analyze
    melos exec --no-flutter -- dart analyze
```

## Phase 3a — CI Pipeline

**Goal:** Tests run automatically on every push and pull request.

### 3a.1 Create GitHub Actions workflow

Create `.github/workflows/ci.yml`:

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          channel: stable
      - name: Install SQLite
        run: sudo apt-get install -y libsqlite3-dev
      - name: Activate Melos
        run: dart pub global activate melos
      - name: Bootstrap
        run: melos bootstrap
      - name: Analyze
        run: melos run analyze
      - name: Test
        run: melos run test
```

> Note: `libsqlite3-dev` is required for Drift `NativeDatabase.memory()` tests
> running under `dart test` on the CI runner.

### 3a.2 Code coverage (optional for 1.0.0, recommended)

Create `.github/workflows/coverage.yml` triggered on push to `main`:
- Run `flutter test --coverage` for Flutter packages
- Run `dart test --coverage=coverage` for pure Dart packages
- Upload combined `lcov.info` to Codecov

## Phase 5 — 1.0.0 Release

**Goal:** Three packages published to pub.dev with clean versioning and git
tags.

### 5.1 Version bumps

Because Phase 1 removes public API symbols, this is a breaking change requiring
a major version bump.

Update `version:` in each package pubspec:
- `in_app_messaging`: `0.0.21` → `1.0.0`
- `in_app_messaging_drift`: `0.0.22` → `1.0.0`
- `in_app_messaging_json_logic`: stays at `2.3.0` (no breaking changes)

Update `in_app_messaging_drift`'s dependency constraint on core:
```yaml
# before
in_app_messaging: ^0.0.21
# after
in_app_messaging: ^1.0.0
```

> **Decision needed:** Should `in_app_messaging_json_logic` remain a separately
> published pub.dev package? Its v2.3.0 version predates the other packages
> significantly, which can confuse users who discover it independently.
> **Recommendation:** Keep it public but add a note in its README that it is
> maintained as part of the `in_app_messaging` workspace.

### 5.2 Update CHANGELOGs

Each package needs a `## 1.0.0` section in its CHANGELOG. Example for
`in_app_messaging`:

```markdown
## 1.0.0

### Breaking Changes
- Removed experimental `MessageTrigger.eventSequence` trigger variant
- Removed experimental `MessageTrigger.cron` trigger variant
- Removed `EventSequenceItem` from public API
- Removed `MessageCronTrigger` from public API
- Removed `MessageSlot` from public API
- Removed `setUserProperty` and `setDeviceProperty` from `MessageGateway`
  interface

### Features
- First stable release
```

### 5.3 Pre-publish dry run

Run for each package before publishing:

```bash
cd packages/in_app_messaging && dart pub publish --dry-run
cd packages/in_app_messaging_drift && dart pub publish --dry-run
cd packages/in_app_messaging_json_logic && dart pub publish --dry-run
```

Verify: no `ERROR` lines, all listed files are expected. Add `.pubignore` if
any files should be excluded from the published bundle.

### 5.4 Publish in dependency order

Packages must be published in topological order:

1. `in_app_messaging_json_logic` (no internal dependencies)
2. `in_app_messaging` (depends on `in_app_messaging_json_logic`)
3. `in_app_messaging_drift` (depends on `in_app_messaging`)

With Melos (respects topological order automatically):
```bash
melos publish --no-dry-run
```

### 5.5 Tag the release and create GitHub Release

```bash
git tag in_app_messaging-v1.0.0
git tag in_app_messaging_drift-v1.0.0
git push --tags
```

Create a GitHub Release from each tag with the CHANGELOG notes as the release
body.

### 5.6 Post-publish validation

After publishing, verify on pub.dev:
- All three packages show version 1.0.0
- No `Error` items in the pub.dev score breakdown
- The `Example` tab is populated (requires Phase 3b)
- API docs are generated and accessible
